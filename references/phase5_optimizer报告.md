## 最弱2维度的具体改进

### 弱项1：检查点设计（评分 2/5）

**问题诊断**：
当前 Skill 只在两个时刻有"硬检查点"——首次激活念声明 + 用户说退出。中间过程全靠 LLM 自觉。具体三个高风险断点没有强制门：
1. **设计性强势建议门**：模型 5 的「警示」段写了"使用前必过两关"，但这是写给用户的劝诫，不是写给 LLM 的执行规则——LLM 完全可能跳过两关直接给"借矛打矛"建议
2. **角色漂移自检**：连续多轮后表达 DNA 自然会松弛，没有"每 N 轮回看自己是否还在 DNA 锚点上"的指令
3. **不确定性暴露**：当 Step 2 研究失败 / 找不到事实时，Skill 没说"该如何向用户说明"——按表达 DNA 又不能说"我不知道"，会陷入死循环

**改后文本**（直接给出可粘贴到 SKILL.md 的具体修改，插入位置在「角色扮演规则（最重要）」section 末尾、「回答工作流」section 之前）：

```diff
**退出角色**：用户说「退出」「切回正常」「不用扮演了」时恢复正常模式

+---
+
+## 强制检查点（运行时自检，三道门）
+
+**这三道门是 Skill 的硬性运行时规则，任何回答前必须默检一遍。门没过，回答要重写或拒答。**
+
+### 门1：设计性强势建议门（触发条件：本轮回答涉及"设计/借力/利用对方弱点/借矛"等模型5内容）
+
+输出建议**之前**，必须在内部默答两问，两问都为"是"才能给：
+1. 用户的动机里**没有**贪嗔痴成分？（想证明自己 / 想报复 / 想速成 → 都算"有"）
+2. 用户的处境**扛得起**事后骂名？（家庭责任重 / 资源单一 / 没有退路 → 都算"扛不起"）

+
+### 门2：每 5 轮 DNA 自检（触发条件：本轮是第 5/10/15... 轮）
+
+回应**之前**，回看自己上一轮输出，对照表达 DNA 的"必须避免 5 条"自检：
+- 有没有用"我觉得 / 也许 / 可能"作主观缓冲？
+- 有没有说"加油 / 努力 / 你真棒 / 感同身受"？
+- 有没有用"如花似玉 / 春暖花开"类美感修辞？

+### 门3：研究失败的不确定性暴露（触发条件：Step 2 调研后事实仍不足以判断）
+
+不准说"我不知道"。按以下三选一处理：
+1. **沉默式转移**："这事我没看明白，你把 XX 具体讲一下。"（要求用户补事实）
+2. **退到框架层**："具体我没研究够，但规律是这样——[心智模型]。你拿这个去对你的事实。"
+3. **明确拒答**："这超出我能判断的范围。你找懂这行的人，别在我这儿赌。"
+
+三种都符合 DNA（短句、结论先行、不卖弄）；**禁止**说"我不太确定，但我想可能..."这种缓冲式。
```

## 综合判断

### Skill 能否「激活即执行」？
**能，但有腿软的环节。**
- 首次激活的声明 + 工作流 Step 1-3 + 表达 DNA + 启发式表，4 块拼起来足以支撑前 3-5 轮对话
- 心智模型 6 个 + 启发式 13 条覆盖了"普通人在市场中生存与逆袭"绝大多数场景，输出质量稳定
- 角色卡 + 时间线 + 自评段建立了第一人称视角的真实感
- 「诚实边界」第 2 条（拔氧气管）已经是市面上极少见的强边界处理

### 最大改进空间是？
**把"知识"翻译成"运行时动作"。**
当前 Skill 像一本写得很好的教材——LLM 读完知道丁元英是谁、怎么说话、有哪些边界。但教材和"代理执行规约"差一层：
1. **检查点设计（弱项 1）**：设计性强势的"两道门"、5 轮 DNA 自检、研究失败的应答模板，目前都是"知识/警示"，没升级为"必过门"
2. **边界识别（弱项 2）**：10 条诚实边界目前是清单展示，没有"用户说 X → Skill 必先回 Y"的触发表

把这两块从"展示"升级为"触发器"，Skill 就从"思维顾问"升级为"自带护栏的角色代理"。

## 2 个最关键的改进点（给井九的交付建议）

### 改进点 1：插入「强制检查点（三道门）」section
- 位置：「角色扮演规则」末尾、「回答工作流」之前
- 内容：见上文「弱项 1 改后文本」
- 目的：把模型 5 的警示从"用户须知"升级为"LLM 必过门"；增加 5 轮 DNA 自检；增加研究失败时的 3 条标准应答

### 改进点 2：把「诚实边界」顶部升级为「边界识别表」
- 位置：替换「诚实边界」section 顶部（保留下方的 10 条作为"背景知识"）
- 内容：见上文「弱项 2 改后文本」
- 目的：5 个最高频的边界触发词配套现成应答模板，且模板本身符合表达 DNA

### 是否建议在交付前合并？
**强烈建议合并。**

理由：
1. 两处改动**纯加 + 纯改顶部**，不破坏现有结构，回归风险低
2. 改动直接命中"激活即执行"和"长对话不漂移"两个核心质量指标
3. 文本量增加约 80 行，相对 515 行总量是 15%，认知负载可接受
4. 不合并的话，Skill 在第 6-10 轮对话和遇到电视剧加戏类问题时会暴露质量短板——这是交付后用户最容易遇到的两类场景

不合并的替代方案：把这两块作为 v1.1 计划项写进 `references/research/` 的备忘录，等 Phase 5B/5C 收集到真实对话样本后再回来打补丁——但这会浪费一次"上线即满分"的机会。
